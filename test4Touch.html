The Failing Tests

  • It checks the user agent. This doesn't really tell you anything about a touch capable laptop, but if it flags ipad/iphone/blackberry/android then touch-only is a safe bet. Hide the mouseover events! Hide them all.
    However, touch-capable desktops exist, and user agents can be faked.
  • This method, btw, is how HULU.com checks whether you're on a phone/tablet or a computer
    So if you'd like to watch Hulu on your phone and don't want to pay for hulu plus, just change your browsers user agent to 'desktop.'
  • It checks for touch capability. On non-IE browsers, if touch is not detected, then it's a pretty safe bet that the user is using a mouse
    (On IE checking pointer types instead)
    However, mobile browsers have been known to throw false negatives, and Desktop browsers, false posatives.
  • It checks the height/width ratio for portrait mode I'm totally lying to you right now, all it does is check if the media querie for protrait mode is flagged. If it's in portrait mode, it's probably a tablet/phone.
    However, portrait mode proves nothing.
    When I turn my MS Surface sideways, I can still use the mouse.
    I can also be really into resizing things.
    That being said, if its less then 800px wide, it's probably phone/tablet (non-HD).
  • Check if touch.events exist The current modernizr solution for touch detection checks if the touch.events exist at all.
    If they do, then we have touch. If not, then no touch.
    However, mouse and touch aren't mutually exclusive!
    Ideally, the UI should respond to each appropriately, but never assume anything.

The Proposed Solution

  • It listens(continually) for touch.events, mouse.events, and MS IE pointer.events If a touch/(non-mouse type pointer)event is detected, enable touch functions and hide your hovers (and other mouse-only functions)
    If a mouse/(pointer.type == mouse) event is detected, do your mouse-only things
  • The draw back is that it must keep listening.
    Not the most efficient thing in the world, but in case your user goes from mouse to touch then back to mouse, you're covered
  • Since webkit/moz touch.events throw mouse events too, a function with a quick timeout blocks the capture of mouse.events immediately after touch.events are fired.
